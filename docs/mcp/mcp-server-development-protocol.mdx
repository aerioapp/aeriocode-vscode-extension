## What Are MCP Servers?

Model Context Protocol (MCP) servers extend AI assistants like Aeriocode by giving them the ability to:

-   Access external APIs and services
-   Retrieve real-time data
-   Control hardware and local systems
-   Perform actions beyond what text prompts alone can achieve

Without MCP, AI assistants are powerful but isolated. With MCP, they gain the ability to interact with virtually any digital or physical system — from flashing firmware on embedded targets to querying requirements databases.

## The Development Protocol

The heart of effective MCP server development is following a structured protocol. This protocol is implemented through a `.aeriocoderules` file that lives at the **root** of your MCP working directory (/Users/your-name/Documents/Aeriocode/MCP).

### Using `.aeriocoderules` Files

A `.aeriocoderules` file is a special configuration that Aeriocode reads automatically when working in the directory where it's placed. These files:

-   Configure Aeriocode's behavior and enforce best practices
-   Switch Aeriocode into a specialized MCP development mode
-   Provide a step-by-step protocol for building servers
-   Implement safety measures like preventing premature completion
-   Guide you through planning, implementation, and testing phases

Here's the complete MCP Server Development Protocol that should be placed in your `.aeriocoderules` file:

````markdown
# MCP Server Development Protocol

⚠️ CRITICAL: DO NOT USE attempt_completion BEFORE TESTING ⚠️

## Step 1: Planning (PLAN MODE)

-   What problem does this tool solve?
-   What API/service will it use?
-   What are the authentication requirements?
    □ Standard API key
    □ OAuth (requires separate setup script)
    □ Other credentials

## Step 2: Implementation (ACT MODE)

1. Bootstrap

    - For web services, JavaScript integration, or Node.js environments:
        ```bash
        npx @modelcontextprotocol/create-server my-server
        cd my-server
        npm install
        ```
    - For data science, ML workflows, or Python environments:
        ```bash
        pip install mcp
        # Or with uv (recommended)
        uv add "mcp[cli]"
        ```

2. Core Implementation

    - Use MCP SDK
    - Implement comprehensive logging
        - TypeScript (for web/JS projects):
            ```typescript
            console.error("[Setup] Initializing server...")
            console.error("[API] Request to endpoint:", endpoint)
            console.error("[Error] Failed with:", error)
            ```
        - Python (for data science/ML projects):
            ```python
            import logging
            logging.error('[Setup] Initializing server...')
            logging.error(f'[API] Request to endpoint: {endpoint}')
            logging.error(f'[Error] Failed with: {str(error)}')
            ```
    - Add type definitions
    - Handle errors with context
    - Implement rate limiting if needed

3. Configuration

    - Get credentials from user if needed
    - Add to MCP settings:

        - For TypeScript projects:
            ```json
            {
            	"mcpServers": {
            		"my-server": {
            			"command": "node",
            			"args": ["path/to/build/index.js"],
            			"env": {
            				"API_KEY": "key"
            			},
            			"disabled": false,
            			"autoApprove": []
            		}
            	}
            }
            ```
        - For Python projects:

            ```bash
            # Directly with command line
            mcp install server.py -v API_KEY=key

            # Or in settings.json
            {
              "mcpServers": {
                "my-server": {
                  "command": "python",
                  "args": ["server.py"],
                  "env": {
                    "API_KEY": "key"
                  },
                  "disabled": false,
                  "autoApprove": []
                }
              }
            }
            ```

## Step 3: Testing (BLOCKER ⛔️)

<thinking>
BEFORE using attempt_completion, I MUST verify:
□ Have I tested EVERY tool?
□ Have I confirmed success from the user for each test?
□ Have I documented the test results?

If ANY answer is "no", I MUST NOT use attempt_completion.
</thinking>

1. Test Each Tool (REQUIRED)
   □ Test each tool with valid inputs
   □ Verify output format is correct
   ⚠️ DO NOT PROCEED UNTIL ALL TOOLS TESTED

## Step 4: Completion

❗ STOP AND VERIFY:
□ Every tool has been tested with valid inputs
□ Output format is correct for each tool

Only after ALL tools have been tested can attempt_completion be used.

## Key Requirements

-   ✓ Must use MCP SDK
-   ✓ Must have comprehensive logging
-   ✓ Must test each tool individually
-   ✓ Must handle errors gracefully
-   ⛔️ NEVER skip testing before completion
````

When this `.aeriocoderules` file is present in your working directory, Aeriocode will:

1. Start in **PLAN MODE** to design your server before implementation
2. Enforce proper implementation patterns in **ACT MODE**
3. Require testing of all tools before allowing completion
4. Guide you through the entire development lifecycle

## Getting Started

Creating an MCP server requires just a few simple steps to get started:

### 1. Create a `.aeriocoderules` file (🚨 IMPORTANT)

First, add a `.aeriocoderules` file to the root of your MCP working directory using the protocol above. This file configures Aeriocode to use the MCP development protocol when working in this folder.

### 2. Start a Chat with a Clear Description

Begin your Aeriocode chat by clearly describing what you want to build. Be specific about:

-   The purpose of your MCP server
-   Which API or service you want to integrate with
-   Any specific tools or features you need

For example:

```plaintext
I want to build an MCP server for the J-Link debug probe.
It should allow me to flash firmware to STM32 targets, read and
write memory registers, and run on-target debugging sessions.
```

### 3. Work Through the Protocol

Aeriocode will automatically start in PLAN MODE, guiding you through the planning process:

-   Discussing the problem scope
-   Reviewing API documentation
-   Planning authentication methods
-   Designing tool interfaces

When ready, switch to ACT MODE using the toggle at the bottom of the chat to begin implementation.

### 4. Provide API Documentation Early

One of the most effective ways to help Aeriocode build your MCP server is to share official API documentation right at the start:

```plaintext
Here's the API documentation for the service:
[Paste API documentation here]
```

Providing comprehensive API details (endpoints, authentication, data structures) significantly improves Aeriocode's ability to implement an effective MCP server.

## Understanding the Two Modes

### PLAN MODE

In this collaborative phase, you work with Aeriocode to design your MCP server:

-   Define the problem scope
-   Choose appropriate APIs
-   Plan authentication methods
-   Design the tool interfaces
-   Determine data formats

### ACT MODE

Once planning is complete, Aeriocode helps implement the server:

-   Set up the project structure
-   Write the implementation code
-   Configure settings
-   Test each component thoroughly
-   Finalize documentation

## Case Study: J-Link Debug Probe MCP Server

Let's walk through the development process of a J-Link MCP server — an aerospace and embedded engineering tool that enables Aeriocode to flash firmware, read memory registers, and debug on-target STM32 microcontrollers directly from the chat.

### Planning Phase

During the planning phase, we:

1. **Defined the problem**: Embedded engineers need to flash firmware, inspect memory, and run debug sessions on STM32 targets without leaving their AI assistant workflow
2. **Selected the interface**: SEGGER J-Link Commander (JLinkExe) CLI for hardware interaction
    - Runs locally via stdio transport (no network authentication needed)
    - Requires J-Link USB connection to the target board
    - Supports all STM32 device families
3. **Designed the tools needed**:
    - Flash firmware binary to target
    - Read memory at a given address
    - Write memory register values
    - Reset the target MCU
    - Read device info (chip ID, core type)
4. **Planned data formatting**:
    - Clean, well-formatted markdown output
    - Hex addresses and register values displayed in tables
    - Success/failure indicators for flash operations
    - Device identification details

### Implementation

We began by bootstrapping the project:

```bash
npx @modelcontextprotocol/create-server jlink-mcp
cd jlink-mcp
npm install
```

Next, we structured our project with:

```plaintext
src/
  ├── commands/
  │   └── jlinkCommander.ts   # J-Link CLI wrapper with error handling
  ├── formatters/
  │   └── registerFormatter.ts # Output formatters for hex/register data
  └── index.ts                 # Main MCP server implementation
```

#### J-Link Command Wrapper

The command wrapper handled communication with the J-Link hardware:

-   **Device connection**: Target detection and connection management
-   **Command execution**: Sending commands via JLinkExe CLI
-   **Error handling**: Robust detection of connection failures and timeouts
-   **Typed interfaces**: Clear TypeScript types for device info and register data

Key implementation details:

```typescript
/**
 * Execute a J-Link commander command and parse the output
 */
private async executeJLinkCommand(commands: string[]): Promise<string> {
  console.error(`[JLink] Executing commands: ${commands.join("; ")}`);
  const args = [
    "-device", this.deviceName,
    "-if", this.interfaceType,
    "-speed", this.speed.toString(),
    "-autoconnect", "1",
    "-CommandFile", "/dev/stdin",
  ];
  // Spawn JLinkExe process and pipe commands
  const result = await this.spawnProcess("JLinkExe", args, commands.join("\n"));
  return result.stdout;
}
```

#### Register Formatting

We implemented formatters to display memory and register data clearly:

```typescript
/**
 * Format memory read results into a markdown table
 */
export function formatMemoryRead(address: number, data: Uint8Array): string {
	const lines: string[] = [`# Memory Read at 0x${address.toString(16).toUpperCase().padStart(8, "0")}\n`];
	lines.push("| Offset | Hex Value | ASCII |");
	lines.push("|--------|-----------|-------|");

	for (let i = 0; i < data.length; i += 16) {
		const chunk = data.slice(i, i + 16);
		const hex = Array.from(chunk).map(b => b.toString(16).padStart(2, "0")).join(" ");
		const ascii = Array.from(chunk).map(b => (b >= 0x20 && b <= 0x7e) ? String.fromCharCode(b) : ".").join("");
		lines.push(`| +0x${i.toString(16).padStart(2, "0")} | ${hex} | ${ascii} |`);
	}

	return lines.join("\n");
}
```

### Challenges and Solutions

During development, we encountered several challenges:

1. **Hardware Connection Reliability**:
    - **Challenge**: J-Link connection dropped intermittently during long operations
    - **Solution**: Implemented connection retry logic with exponential backoff and automatic reconnection
2. **Output Parsing**:
    - **Challenge**: JLinkExe output format varies across firmware versions and device families
    - **Solution**: Created device-specific output parsers with regex-based extraction and fallback patterns
3. **Concurrent Access**:
    - **Challenge**: Multiple tools trying to use the J-Link simultaneously caused conflicts
    - **Solution**: Implemented a command queue with mutex locking to serialize J-Link operations

### Lessons Learned

Our J-Link MCP server implementation taught us several key lessons:

1. **Hardware Abstraction**: Wrap hardware CLI tools with clean interfaces — don't expose raw command syntax to the MCP layer
2. **Defensive Parsing**: Hardware tools produce inconsistent output; always parse defensively with fallbacks
3. **Serialize Hardware Access**: Debug probes are single-connection devices; implement proper queuing
4. **Test with Real Hardware**: Simulators miss connection and timing issues that only appear on-target
5. **Safety First**: Flash and write operations should require explicit user approval by default

## Core Implementation Best Practices

### Comprehensive Logging

Effective logging is essential for debugging MCP servers:

```typescript
// Start-up logging
console.error("[Setup] Initializing J-Link MCP server...")

// Hardware command logging
console.error(`[JLink] Flashing firmware: ${filePath} to ${address}`)

// Error handling with context
console.error(`[Error] Tool execution failed: ${error.message}`)

// Connection state
console.error(`[Connection] Connected to ${deviceName} via ${interfaceType}`)
```

### Strong Typing

Type definitions prevent errors and improve maintainability:

```typescript
export interface JLinkConfig {
	deviceName: string
	interfaceType: "SWD" | "JTAG"
	speed: number
	serialNumber?: string
}

/**
 * Validate that a hex address is properly formatted
 */
function validateHexAddress(address: unknown): asserts address is string {
	if (typeof address !== "string" || address.trim() === "") {
		throw new McpError(ErrorCode.InvalidParams, "A hex address is required")
	}

	// Validate hex format (0x prefix, valid hex chars)
	const hexRegex = /^0x[0-9A-Fa-f]+$/;
	if (!hexRegex.test(address)) {
		throw new McpError(ErrorCode.InvalidParams, `Invalid hex address: ${address}`)
	}
}
```

### Graceful Error Handling

Implement robust error handling that maintains a good user experience:

```typescript
try {
	switch (request.params.name) {
		case "flash_firmware": {
			// Implementation...
		}

		// Other cases...

		default:
			throw new McpError(ErrorCode.MethodNotFound, `Unknown tool: ${request.params.name}`)
	}
} catch (error) {
	console.error(`[Error] Tool execution failed: ${error instanceof Error ? error.message : String(error)}`)

	if (error instanceof McpError) {
		throw error
	}

	return {
		content: [
			{
				type: "text",
				text: `Error: ${error instanceof Error ? error.message : String(error)}`,
			},
		],
		isError: true,
	}
}
```

## MCP Resources

Resources let your MCP servers expose data to Aeriocode without executing code. They're perfect for providing context like files, API responses, or database records that Aeriocode can reference during conversations.

### Adding Resources to Your MCP Server

1. **Define the resources** your server will expose:

```typescript
server.setRequestHandler(ListResourcesRequestSchema, async () => {
	return {
		resources: [
			{
				uri: "file:///project/readme.md",
				name: "Project README",
				mimeType: "text/markdown",
			},
		],
	}
})
```

2. **Implement read handlers** to deliver the content:

```typescript
server.setRequestHandler(ReadResourceRequestSchema, async (request) => {
	if (request.params.uri === "file:///project/readme.md") {
		const content = await fs.promises.readFile("/path/to/readme.md", "utf-8")
		return {
			contents: [
				{
					uri: request.params.uri,
					mimeType: "text/markdown",
					text: content,
				},
			],
		}
	}

	throw new Error("Resource not found")
})
```

Resources make your MCP servers more context-aware, allowing Aeriocode to access specific information without requiring you to copy/paste. For more information, refer to the [official documentation](https://modelcontextprotocol.io/docs/concepts/resources).

## Common Challenges and Solutions

### Hardware Connection Issues

**Challenge**: Hardware debug probes can disconnect unexpectedly or fail to connect.

**Solution**:

-   Implement connection retry logic with exponential backoff
-   Provide clear error messages about connection state
-   Auto-detect when the target is not powered or not responding

### Missing or Limited CLI Features

**Challenge**: Hardware CLI tools may not expose all the functionality you need.

**Solution**:

-   Combine multiple CLI commands to achieve the desired operation
-   Use raw memory read/write to implement custom register-level operations
-   Fall back to J-Link scripting for advanced automation

### Concurrent Hardware Access

**Challenge**: Debug probes are single-connection devices that don't support concurrent access.

**Solution**:

-   Implement a command queue to serialize operations
-   Use mutex locking to prevent concurrent J-Link sessions
-   Provide clear error messages when the probe is busy
