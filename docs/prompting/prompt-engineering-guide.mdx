## .aeriocodeignore File Guide

### Overview

The `.aeriocodeignore` file is a project-level configuration file that tells Aeriocode which files and directories to ignore when analyzing your codebase. Similar to `.gitignore`, it uses pattern matching to specify which files should be excluded from Aeriocode's context and operations.

### Purpose

-   **Reduce Noise**: Exclude auto-generated files, build artifacts, and other non-essential content
-   **Improve Performance**: Limit the amount of code Aeriocode needs to process
-   **Focus Attention**: Direct Aeriocode to relevant parts of your codebase
-   **Protect Sensitive Data**: Prevent Aeriocode from accessing sensitive configuration files

### Example .aeriocodeignore File

```
# Build outputs

dist/
out/

# Compiled objects and binaries

# Environment and credentials

# Large data and model files

```

## Prompting Aeriocode 💬

**Prompting is how you communicate your needs for a given task in the back-and-forth chat with Aeriocode.** Aeriocode understands natural language, so write conversationally.

Effective prompting involves:

-   Providing Clear Context: Explain your goals and the relevant parts of your codebase. Use `@` to reference files or folders.
-   Breaking Down Complexity: Divide large tasks into smaller steps.
-   Asking Specific Questions: Guide Aeriocode toward the desired outcome.
-   Validating and Refining: Review Aeriocode's suggestions and provide feedback.

### Prompt Examples

#### Context Management

-   **Starting a New Task:** "Aeriocode, let's start a new task. Create `flight_controller.c`. We need to implement a PID control loop for the pitch axis autopilot. Here are the requirements…"
-   **Summarizing Previous Work:** "Aeriocode, summarize what we did in the last air data computer integration task. I want to capture the sensor fusion algorithm changes and outstanding verification issues. Save this to `aeriocode_docs/adc-integration-summary.md`."

#### Debugging

-   **Analyzing an Error:** "Aeriocode, I'm getting a hard fault exception during the sensor read ISR on the STM32F4. The stack trace points to `imu_driver.c`. Analyze this fault and suggest a fix."
-   **Identifying the Root Cause:** "Aeriocode, the telemetry downlink drops packets when the data rate exceeds 115200 baud. The issue might be in the UART DMA buffer management or the packet framing logic. Help me find the root cause."

#### Refactoring

-   **Improving Code Structure:** "Aeriocode, the `navigation_solution()` function in `ins.c` is over 500 lines and handles both Kalman filter prediction and measurement updates. Refactor it into separate prediction and update modules."
-   **Simplifying Logic:** "Aeriocode, the attitude quaternion normalization logic in `attitude_estimation.cpp` uses nested conditionals that are hard to verify. Simplify the logic and add unit tests for each branch."

#### Feature Development

-   **Brainstorming New Features:** "Aeriocode, I want to add a geofencing feature to the ground control station that triggers failsafe actions when the UAV leaves the defined airspace. Brainstorm some approaches and consider DO-178C DAL-C implications."
-   **Generating Code:** "Aeriocode, create a sensor health monitoring module in C that tracks IMU, GPS, and barometer status. It should use majority voting for redundancy and trigger an alert when any sensor exceeds its error threshold. Generate the code with MISRA-C compliant style."

## Advanced Prompting Techniques

-   **Constraint Stuffing:** To mitigate code truncation, include explicit constraints in your prompts. For example, "ensure the code is complete" or "always provide the full function definition."
-   **Confidence Checks:** Ask Aeriocode to rate its confidence (e.g., "on a scale of 1-10, how confident are you in this solution?")
-   **Challenge Aeriocode's Assumptions:** Ask “stupid” questions to encourage deeper thinking and prevent incorrect assumptions.
